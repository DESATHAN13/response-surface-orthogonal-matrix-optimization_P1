DOCUMENT SYSTEM: RESPONSE SURFACE METHODOLOGY (RSM) SEARCH PIPELINE
LANGUAGE: MATLAB

================================================================================
1. EXECUTIVE SUMMARY
================================================================================
This MATLAB script functions as a high-performance optimization and search pipeline engineered specifically to construct statistically efficient experimental designs within Response Surface Methodology (RSM). At its operational core, the script executes a deterministic, brute-force grid search over a highly constrained multi-dimensional parameter space to identify globally optimal design layouts. 

Candidate matrices are cross-evaluated using dual mathematical pathways:
  * Statistical Efficiency: Evaluated via D-value criteria to minimize parameter estimation variance.
  * Geometric Uniformity: Evaluated via rotatability measures (Q* criterion) to ensure balanced predictive variance in all directions from the design center.

================================================================================
2. COMPONENT BREAKDOWN
================================================================================

2.1 PARAMETRIC ORTHOGONAL DESIGN (OD) GENERATORS
The terminal initialization sequence of the pipeline constructs a suite of anonymous matrix-mapping functions (designated as X2, X4, X6a, X8, X12a, etc.). These map scalar variables to rigid Orthogonal Design (OD) algebraic frameworks ranging in dimension from 2x2 up to 20x20.
  * Inputs: Scalar weight metrics (a, b, c, d, etc.).
  * Outputs: Dimensionally stable matrix coordinate layouts consisting of zero-elements, positive shifts, or negative reflections.
  * Purpose: These functions serve as the multi-variable axial components (A or T_OD) embedded directly into the response surface layout. Unlike classical Central Composite Designs (CCDs) that vary a single factor per axial run, these generators force multiple inputs to transition simultaneously, drastically reducing standalone run counts while boosting interaction sensitivity.

2.2 EXPERIMENTAL PARAMETERS & GLOBAL FILE INITIALIZATION
The script explicitly establishes structural limits and execution constraints for the active search matrix:
  * Run Constraints (n): Fixed at 96 total experimental runs.
  * Factor Mapping (m, s): Active factor space (m) and target column-selection subset (s) are both locked to a 10-dimensional matrix layout.
  * File Logging Operations: The script dually instantiates three target text files (.txt) on disk to stream and log matching matrix records. These records are broken into distinct evaluation branches:
    - Branch 1: 'Evang' (Tracking optimizations according to the Evangelaras criterion).
    - Branch 2: 'Nguyen' (Tracking optimizations according to the Nguyen criterion).
    - Branch 3: 'Q3ge' (Tracking optimizations maximizing geometric Rotatability).

2.3 FACTORIAL BLOCK ASSEMBLIES
To ground the full second-order polynomial space, the framework aggregates a permanent, highly structured fractionated factorial foundation (F). This foundation is synthesized through two independent sub-blocks:
  * Sub-Block A (Xf1): A core 32x32 Hadamard matrix. The script applies nchoosek combination logic to systematically isolate and map optimal column variations.
  * Sub-Block B (C10 / f10): A specialized 10x10 conference matrix layout combined vertically with its inverted counterpart (-1 * C10).
  * Assembly Matrix (Xf): Both arrays are vertically concatenated via the matrix assignment Xf = [Xf2; f10] to form the permanent, immutable base coordinates of the design.

2.4 NESTED SEARCH ENGINE LOOPS
The script deploys an extensive array of nested programmatic loops (for j1 = 1:..., for j2 = 1:...).
  * Operation: The engine iterates step-wise through binary coordinate weights (0 or 1) assigned across active structural variants.
  * Real-Time Compilation: During every loop step, variables are mapped into dynamic structural combinations, such as compiling block-diagonal configurations like XT22 = [XT20, zeros(20,2); zeros(2,20), X2(...)]. The resulting composite layout is immediately isolated and designated as the active axial matrix (XA).

2.5 DESIGN CONSTRUCTION & CRITERIA EVALUATION
For each active loop iteration, the compiled structural data is passed down to the 'Construction3g2' and 'Construction3g' execution modules, which process the inputs through four distinct mathematical phases:

  Phase 1: Matrix Composition
  The total candidate design layout is generated by vertically nesting the structural parts along with a designated block of center runs: D = [Xf; T; C; -T].

  Phase 2: Full Second-Order Model Expansion
  The script passes the design matrix D to the local function 'MakeModelMatrix(D, 'full')'. This programmatically expands the layout into a complete second-order polynomial model matrix (Xm), constructing explicit columns for:
    - The intercept column vector (ones).
    - Linear factor terms (Xi).
    - Pure quadratic factor terms (Xi^2).
    - Two-factor cross-product interaction terms (Xi * Xj).

  Phase 3: Information Determinant Extraction
  The script extracts the raw algebraic information matrix via the transpose cross-product (Xm' * Xm) and computes its absolute determinant.

  Phase 4: Optimization Threshold Checking
  The determinant is normalized against the total calculated parameter size (p) and run size (nn). The resulting metric is checked directly against running performance boundaries (Dbest1, Dbest2, Qbest). If the matrix scores higher than a historical threshold and stays above the critical safety value (critical_D), the search loop updates the record and triggers 'WriteDesignToFile' to write the design matrix to disk.
